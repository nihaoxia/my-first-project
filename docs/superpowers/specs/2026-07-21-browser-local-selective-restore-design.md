# 浏览器本地备份选择性恢复设计

日期：2026-07-21

状态：书面规格已批准

依赖规格：`docs/superpowers/specs/2026-07-21-browser-local-data-backup-design.md`

## 1. 背景

Stray Pages 已支持把当前账号作用域中的六类浏览器长期数据加密为版本 1 `.spbackup` 文件，并在认证、数据校验和明确确认后整体替换恢复。六类数据为：

1. 本地原书；
2. 本地译本；
3. 词汇；
4. 句子；
5. 笔记；
6. 阅读器收藏。

当前整体替换适合完整迁回同一账号，但用户无法只恢复误删的学习数据而保留其他当前数据。本阶段在不改变备份格式、加密边界和同账号限制的前提下，增加按分类选择恢复范围的能力。

## 2. 目标

- 用户检查备份成功后，可以选择需要恢复的数据分类。
- 选中的分类按备份内容整体替换；未选中的分类保持原样。
- 本地原书与本地译本作为一个不可拆分的关联组，避免产生悬空译本引用。
- 只读取、写入和回滚选中的存储键。
- 选择范围、账号作用域和确认状态在执行前再次验证。
- 保留现有零上传、零新增依赖、零收费资源和浏览器本地加密边界。

## 3. 非目标

本阶段不实现：

- 记录级选择；
- 按记录 ID 自动合并；
- 冲突比较、冲突解决器或历史版本选择；
- 跨账号迁移或跨账号恢复；
- 云端同步、云端自动备份或定时备份；
- 修改 `.spbackup` 格式、版本、KDF、加密算法、AAD 或大小预算；
- 在备份文件中保存恢复选择；
- 恢复上传草稿、云端导入标记或六类范围之外的浏览器数据；
- 新增 npm 依赖、网络请求、模型调用或云资源。

“选择性恢复”只表示按分类选择后整体替换，不表示合并。

## 4. 设计选择

### 4.1 采用方案 A：固定恢复组

恢复范围使用五个固定组：

| 恢复组 | 覆盖的数据键 | 说明 |
| --- | --- | --- |
| 原书与译本 | 本地原书、本地译本 | 不可拆分，始终同时读取、替换和回滚 |
| 词汇 | 词汇 | 独立选择 |
| 句子 | 句子 | 独立选择 |
| 笔记 | 笔记 | 独立选择 |
| 阅读器收藏 | 阅读器收藏 | 独立选择 |

界面仍展示六类数据的独立计数，但恢复选择使用五个复选框。“原书与译本”复选框旁同时显示两项计数。这样明确表达关联约束，不提供看似可选但会被自动改写的两个复选框。

### 4.2 未采用的方案

- 不允许原书与译本任意拆选。该方案必须把当前数据与备份数据组合后重新验证引用关系，状态随当前数据变化，错误面和测试矩阵更大。
- 不按记录 ID 合并。同 ID 不同内容、删除语义、顺序、译本引用和收藏重复都需要新的冲突规则，超出本阶段范围。

## 5. 恢复选择模型

备份文件保持版本 1，不增加字段。选择只存在于检查成功后的当前页面内存中。

恢复核心新增固定组类型：

```ts
type LocalBackupRestoreGroup =
  | "library"
  | "vocabulary"
  | "sentences"
  | "notes"
  | "readerSelections";
```

恢复调用必须显式传入 `selectedGroups`。核心不依赖界面传入的顺序，而是把选择映射回现有六键权威顺序：

1. 本地原书；
2. 本地译本；
3. 词汇；
4. 句子；
5. 笔记；
6. 阅读器收藏。

`library` 一次映射为本地原书和本地译本两个连续目标。重复组、未知组、空选择和非数组输入均返回稳定的 `INVALID_SELECTION`，并且零读取、零写入。

界面使用上述五个固定组，不接受自由字符串。候选失效时同时清除选择状态。

## 6. 页面交互

### 6.1 检查前

文件选择、口令输入、扩展名和大小预检、解密、同账号判断、六类数据校验均沿用现有流程。检查阶段仍然零写入。

### 6.2 检查成功后

检查成功后显示：

- 创建时间；
- 六类数据计数；
- “选择恢复内容”区域；
- 五个恢复组复选框；
- 明确确认框；
- “恢复所选数据”按钮；
- “取消”按钮。

五个恢复组默认全部选中，以保持现有完整恢复的默认语义。用户可以取消不需要恢复的组。

任何恢复组变化都会：

- 立即取消确认框；
- 清除之前的恢复结果提示；
- 保留已检查候选和文件；
- 更新按钮的可用状态。

当没有选中任何组时，确认框和恢复按钮禁用，并显示“请至少选择一类要恢复的数据。”。用户重新选中任一组后可以继续。

确认文案固定为：“我了解恢复会替换所选分类的当前本地数据”。按钮文案固定为：“恢复所选数据”；执行中为“正在恢复”。

### 6.3 候选失效

以下情况清除候选、选择和确认：

- 选择了另一个文件；
- 取消恢复；
- 恢复执行结束，无论成功或失败；
- 当前页面卸载；
- 再次检查文件；
- 执行时发现账号作用域变化。

候选建立时恢复组重置为五组全选，不沿用上一份文件的选择。

## 7. 恢复事务

恢复事务顺序固定如下：

1. 验证 `sourceScopeFingerprint`、检查时作用域和执行时作用域三者相同；
2. 验证恢复组输入合法且至少选择一组；
3. 按六键权威顺序展开选中的目标键；
4. 完整读取所有选中键的当前快照；
5. 若任一快照读取失败，返回 `READ_FAILED`，不执行任何写入；
6. 按权威顺序写入选中键；备份中为空的选中分类删除当前键；
7. 若任一写入失败，停止后续主写入；
8. 把失败键也视为已尝试键，并按相反顺序回滚所有已尝试的选中键；
9. 一个回滚失败后继续尝试其他回滚；
10. 返回成功、完整回滚失败结果或不完整回滚结果。

未选中的键在整个事务中必须零读取、零写入、零删除、零回滚。它们的值和是否存在都保持不变。

作用域验证先于选择验证，二者都先于浏览器存储访问。这样账号变化始终优先 fail closed，非法选择也不会触碰数据。

## 8. 结果与错误文案

沿用现有文件、认证、数据、作用域、读取、写入和回滚错误文案，只调整成功与写入结果以明确“所选数据”：

- 成功：“所选本地数据已恢复。请刷新页面，让相关工作区重新读取数据。”
- 写入失败且回滚完成：“恢复失败，所选分类的原有本地数据已恢复，未完成替换。”
- 回滚失败：“恢复失败，且无法完整还原所选分类的原有本地数据。请不要继续编辑，并保留备份文件。”
- 选择为空：“请至少选择一类要恢复的数据。”
- 选择结构非法：普通界面不生成该状态；若核心返回 `INVALID_SELECTION`，显示“恢复范围无效，未写入任何内容。”
- 作用域不同：“此备份来自另一个账号，当前账号不能恢复。”
- 当前读取失败：“无法读取当前账号的本地数据，未开始恢复。”

普通用户文案不出现底层键名、事务、KDF、PBKDF2、AES、AAD、Provider、token、API 或审计术语。

## 9. 安全、隐私与费用边界

- 不修改现有 Web Crypto 和文件解析实现。
- 不把选择写回备份文件、localStorage、Cookie、URL、日志或分析服务。
- 不把备份、口令、预览、选择或恢复结果上传。
- 不枚举 `localStorage`；只使用选中组映射出的固定作用域键。
- 不读取未选中键，即使读取仅用于验证也不允许。
- 不发送网络请求，不调用 EdgeOne、Blob、KV、Models、COS、TTS 或其他云服务。
- 不新增依赖，不创建收费资源，不改变零费用门禁。

## 10. 可访问性与布局

- “选择恢复内容”使用带可见标题的 `<fieldset>` 和 `<legend>`。
- 每个恢复组使用原生复选框和可见 label。
- 原书与译本关联说明通过可见文字和 `aria-describedby` 提供，不只依赖颜色或图标。
- 空选择提示使用 `role="status"`；恢复错误继续使用 `role="alert"`。
- 键盘可以完成选择、确认、恢复和取消。
- 延续现有内联卡片；不新增弹窗、步骤向导或新设计系统。
- 窄屏下复选框自然单列，桌面可使用两列，但不得产生横向滚动。

## 11. 测试策略

### 11.1 纯逻辑测试

- 五个恢复组准确映射为六个固定数据键；
- `library` 始终同时映射原书和译本；
- 输入顺序不改变权威执行顺序；
- 重复组、未知组、空选择和非数组输入均稳定拒绝；
- 非法选择零读取、零写入；
- 只读取和写入选中键，未选中键完全不接触；
- 选中空分类会删除对应键；
- 每个选中主写入位置失败时，已尝试键均按反向顺序回滚；
- 原始缺失键回滚为缺失；
- 一个回滚失败后仍继续其余回滚；
- 作用域变化优先于选择验证且零存储访问；
- 全选五组的结果与现有整体恢复一致。

### 11.2 UI 合同测试

- 检查成功后才显示恢复范围；
- 五组默认全选；
- 原书和译本只有一个关联组控件，但分别显示计数；
- 更改选择会取消确认；
- 空选择禁用确认和恢复；
- 恢复调用显式传入选中组；
- 文件变化、取消和恢复结束清除选择；
- 确认和按钮使用“所选分类/所选数据”文案；
- 不出现自动合并、跨账号恢复或上传行为；
- 不增加网络调用和存储枚举。

### 11.3 回归与验收

- 现有加密、解密、严格文件解析、同账号拒绝和整体恢复回归全部通过；
- 全量 `pnpm test`、`pnpm lint`、`pnpm typecheck`、`pnpm build`、`pnpm verify:zero-cost` 和 `git diff --check` 全部通过；
- 浏览器桌面与窄屏验收覆盖默认全选、取消部分组、空选择、确认重置、成功恢复和错误提示；
- 扫描确认无网络请求、云 SDK、存储枚举、敏感日志或新增依赖。

## 13. 完成定义

- 用户可以在检查成功后选择五个固定恢复组；
- 原书和译本始终作为一个不可拆分组；
- 默认全选保持现有完整恢复体验；
- 至少选择一组且再次明确确认后才能恢复；
- 只接触选中键，未选中数据保持原样；
- 选中键整体替换、空分类删除和失败回滚均有穷举测试；
- 同账号、零上传、无自动合并、无新依赖和零费用边界保持不变；
- 文档准确说明“按分类整体替换”而非“合并”；
- 本地全量门禁、扫描、远端 SHA 和 GitHub Actions 全部成功。
