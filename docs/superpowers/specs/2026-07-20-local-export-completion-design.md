# 本地导出闭环设计

## 背景

Stray Pages 已能生成译本 TXT、词汇 CSV 和句子 Markdown 的内容，并且词汇本与句子本已接入浏览器下载。当前仍有三个彼此相关的断点：完整译本 TXT 没有用户可触发的真实下载入口，笔记本没有 Markdown 导出，现有下载 DOM 操作只存在于词汇/句子专用组件中。路线图和 README 还保留“真实浏览器下载尚未接入”及旧生产架构描述，与当前代码不一致。

本子项目只完成无需 EdgeOne、无需网络、无需新增依赖的文本导出闭环。EdgeOne 开通、EPUB/MOBI/PDF、AI 问答、语音朗读和任何付费服务均不在本次范围内。

## 目标与成功标准

完成后，用户可以：

1. 从本地译本或云端译本阅读器下载当前可读章节组成的完整 UTF-8 TXT；
2. 从本地或云端笔记本下载全部可导出的笔记 Markdown；
3. 继续使用现有词汇 CSV 和句子 Markdown 下载，且它们复用同一个下载组件；
4. 在浏览器不支持下载或下载准备失败时看到明确错误，而不是错误地显示成功；
5. 在整个导出过程中不发起额外网络请求、不写入云端、不创建对象存储对象，也不触发模型调用。

验收要求包括单元测试、静态 UI 合约测试、全量测试、Lint、类型检查、生产构建和文档一致性检查。

## 方案选择

采用“统一文本下载组件 + 既有导出构建器”的方案。

- 不在每个页面复制 `Blob`、对象 URL 和临时链接逻辑；下载副作用集中到一个客户端组件。
- 保留 `src/lib/export/translation-export.ts` 和 `src/lib/export/study-export.ts` 作为纯内容构建层，Node 测试不依赖 DOM。
- 词汇、句子、笔记和译本只负责向统一组件提供文件名、内容、MIME 类型和按钮文案。
- 不引入 zip、EPUB 或文件保存第三方包，避免把本子项目扩展为格式解析项目。

没有采用“新增下载 API 路由”，因为文本内容已经在页面或服务端读取流程中可用，额外路由会增加鉴权、网络失败和重复查询面。也没有采用“继续为每个页面保留专用 DOM 下载代码”，因为该方案会重复错误处理并扩大行为漂移风险。

## 架构与组件边界

### 纯导出构建层

`src/lib/export/translation-export.ts` 继续负责稳定章节顺序、标题头和 TXT 内容。它新增或复用明确的 UTF-8 文本下载元数据，但不访问浏览器 API。

`src/lib/export/study-export.ts` 新增笔记 Markdown 构建器。笔记按当前服务返回的稳定顺序导出，每条包含标题、来源、更新时间和正文；Markdown 控制字符按既有文本导出原则保留为普通内容，不执行 HTML。

所有纯导出构建器必须返回非空安全文件名。现有标题转文件名逻辑无法从任意 Unicode 标题得到拉丁字符时，统一使用明确的 `stray-pages-export` 回退名；笔记本固定使用 `stray-pages-notes.md`，避免把多来源笔记错误归到某一本书。

`src/lib/cloud/study-core.ts` 的有界全量导出读取扩展到 `note`。云端词汇、句子和笔记均最多导出 10,000 条；到达上限且仍有下一页时拒绝导出并显示范围过大的提示，避免无界内存使用。

### 统一浏览器下载组件

新增通用客户端文本下载组件，职责只有：

1. 接收已经构建完成的 `fileName`、`content`、`mimeType` 和 `label`；
2. 创建 `Blob` 和对象 URL；
3. 创建带 `download` 属性的临时链接并触发一次点击；
4. 无论成功或异常都清理临时链接和对象 URL；
5. 只有完整完成上述步骤后才显示“已准备下载”，异常时显示“无法准备下载，请重试”。

组件不接受 URL，不调用 `fetch`，不读取 Cookie、Local Storage 或云端配置。文件名由纯导出构建层产生，组件只拒绝空文件名、路径分隔符和控制字符，不自行改变业务标题。

现有 `StudyExportButton` 将改为薄适配层或被统一组件替代，词汇和句子行为保持不变。

### 译本数据流

本地译本父组件已经持有完整的本地译本记录；它从所有可读章节构造 TXT 导出输入，并把结果传给阅读器。云端阅读器服务端组件已经一次取得所有可读译文章节；它在服务端构造同样的导出输入后，把文本下载数据作为可序列化属性传给客户端阅读器。

阅读器只在至少存在一个可读章节时显示“下载完整译本 TXT”。只导出当前账号已经有权读取且已经完成的章节，不额外读取原文对象，不包含失败、取消或尚未生成内容的章节。

### 笔记数据流

本地模式使用当前账号作用域内的全部有效笔记构建 Markdown。云端模式使用与词汇/句子相同的有界分页扫描读取所有笔记，再构建 Markdown。云端读取失败时页面明确显示导出不可用，不回退到本地或示例数据。

为避免在用户编辑但尚未保存时导出与界面不一致的内容，笔记导出只包含已保存数据；未点击“保存”的草稿不进入文件。

## 错误处理与安全边界

- 空文件名、包含 `/`、`\\` 或控制字符的文件名在下载前拒绝。
- 文本内容允许为空；空笔记本仍可导出带标题的 Markdown，译本必须至少包含一个可读章节。
- 对象 URL 在成功和失败路径中均回收，临时链接在点击后立即移除。
- 浏览器缺少 `Blob`、`URL.createObjectURL` 或 DOM 能力时显示失败提示，不抛出未处理异常。
- CSV 保持 `text/csv;charset=utf-8`，Markdown 保持 `text/markdown;charset=utf-8`，译本使用 `text/plain;charset=utf-8`。
- 导出内容不作为 HTML 插入页面；Markdown 和 TXT 中的用户文本仅作为下载字节保存。
- 本子项目不会修改云端写入门禁、额度账本或生产环境变量，也不会创建任何外部资源。

## 测试策略

严格使用 TDD：先添加失败测试并确认失败原因是功能缺失，再写最小实现。

1. 纯逻辑测试覆盖笔记 Markdown 的标题、顺序、来源、更新时间、正文和稳定文件名。
2. 下载元数据测试覆盖译本 TXT、词汇 CSV、句子 Markdown、笔记 Markdown、Unicode 标题回退名、非法路径文件名和用户提示。
3. 云端学习数据测试覆盖笔记分页全量扫描、10,000 条上限和超限拒绝。
4. UI 合约测试固定本地/云端译本都提供 TXT 下载数据，笔记页提供 Markdown 下载，词汇/句子仍使用统一下载组件。
5. 用户文案测试禁止继续把真实文本下载描述为“尚未接入”。
6. 完成后运行 `pnpm test`、`pnpm lint`、`pnpm typecheck`、`pnpm build`、`pnpm verify:zero-cost` 和 `git diff --check`。

## 文档更新

README 将以 EdgeOne 零费用架构作为当前生产方向，保留 Supabase/CVM/COS/短信内容时明确标记为历史兼容方案，不能再将其写成当前生产要求。当前能力表将把译本 TXT、词汇 CSV、句子 Markdown 和笔记 Markdown 标记为真实浏览器下载。

路线图阶段 8 的“真实浏览器下载尚未接入”在实现完成后改为已完成，并保留真实 EPUB 打包为独立后续项目。开发日志记录本轮红灯、绿灯、全量验证和提交证据。

## 明确不在范围内

- EdgeOne Makers 开通、项目创建、Blob Store 创建和真实部署；
- EPUB 导入、EPUB 打包、MOBI/PDF 解析；
- AI 问答、模型 Key、联网查证或任何模型调用；
- 语音朗读；
- 未保存笔记草稿导出；
- 服务器生成下载文件、持久化导出文件或导出审计记录；
- 对历史 Supabase/Prisma 兼容代码进行无关重构。

这些项目在本地导出闭环完成后分别进入独立的规格、计划和实现周期。
